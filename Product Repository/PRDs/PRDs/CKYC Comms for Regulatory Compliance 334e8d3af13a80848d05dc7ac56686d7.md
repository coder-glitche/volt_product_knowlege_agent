# What problem are we solving?

- Currently, DSP Finance (via Decentro) will search for an existing CKYC record in the CERSAI registry (using PAN) → if found, it downloads the data, compares it, and uploads any updates → if not found, it creates a new CKYC record.
- Regulatory compliance mandates that customers be notified when a CKYC record is created by DSP Finance.
- Currently, there is no automated mechanism to trigger an acknowledgement SMS to customers upon CKYC ID creation.
- Without this flow, DSP Finance risks non-compliance with RBI regulatory notification norms.

---

# How do we measure success?

- **Send accuracy** — 100% of customers receive an SMS upon CKYC id creation
- **log coverage** — 100% of  records logged with per-record dispatch status for compliance audit.

---

# What is the solution?

## Overview

- In the Decentro bulk web-hook response, `ckycReferenceId` indicates an existing CERSAI CKYC record; if it is null, it means a new CKYC ID was created for the customer which is the regulatory trigger for sending the sms.
- CKYC uploads are processed in batches of ~40 records per API call; within each batch, only a subset of records are created for first-time (i.e. `ckycReferenceId: null` and `status: COMPLETED`  at record level in the web-hook response).

## In Scope

- Automated SMS trigger inline within the Decentro web-hook processing flow
- Per-record eligibility check (status == COMPLETED AND ckycReferenceId == null)
- Customer phone number resolution from internal system using fxcid_key
- Dynamo DB logging at per-record level for all outcomes (success, failure)
- Trigger Condition Logic
    - When Decentro’s bulk CKYC web-hook is received, the system inspects each record in the `data` object.
    - For every record where:
        - `status` = `COMPLETED`  AND
        - `additionalData.ckycReferenceId` = `null`
    - the system automatically dispatches an acknowledgement SMS to the customer with their newly generated CKYC ID.
    
    ```
    FOR each fxcid_key IN webhook.data:
        record = webhook.data[fxcid_key]
    
        IF record.status == "COMPLETED"
        AND (record.additionalData.ckycReferenceId == null
        AND record.additionalData.ckycId IS NOT null):
        
            → trigger CKYC acknowledgement SMS for this customer
        ELSE:
            → no comms triggered
    ```
    

## Out of Scope

- Email channel — SMS only for CKYC acknowledgement
- Retry on record-level SMS delivery failure — logged as failed, no retry
- Live delivery tracking (click/open events)

# Communication Specification

### Channel

- SMS only

### Required Fields

| Field | Value |
| --- | --- |
| DLT Template ID | 1107177641696897592 |
| SMS Body Template | Dear {#alphanumeric#}, your CKYC identifier - {#alphanumeric#} is generated by DSP Finance and is linked to your Customer ID: {#alphanumeric#} |
| variables | customer_name, ckyc_id, client_id |

### Variables

| Variable Name | Source | Description |
| --- | --- | --- |
| customer_name | internal DB | name of the customer |
| ckyc_id | `webhook.data[fxcid_key].additionalData.ckycId` | Newly generated CKYC identifier from Decentro response |
| customer_id | Derived from system using `fxcid_key` (fenix client ID / opportunity ID) | DSP's internal customer identifier linked to the CKYC record |

### Web-hook Reference

#### Source

Decentro Bulk CKYC Individuals Upload — webhook callback

#### Key Fields Used

| Field Path | Type | Description |
| --- | --- | --- |
| `type` | string | Must be `BULK_CKYC_INDIVIDUALS_UPLOAD` to identify the webhook type |
| `status` (batch level) | string | Overall batch status — `COMPLETED` |
| `data` | object | Map of `fxcid_key → record` for each customer in the batch |
| `data[fxcid].status` | string | Record-level status — `COMPLETED` / `FAILED` |
| `data[fxcid].additionalData.ckycId` | string | The newly generated CKYC ID for the customer |
| `data[fxcid].additionalData.ckycReferenceId` | string / null | `null` = first-time CKYC creation; non-null = existing record update |
| `data[fxcid].stage` | string | e.g. `RECORD_UPLOAD_SUCCESSFUL`, `RECORD_VALIDATION_FAILED` |
| `bulkDecentroTxnId` | string | Decentro's unique transaction ID for the batch |
| `referenceId` | string | DSP's reference ID for the batch |
| `summary.processedRecords` | int | Total records processed in this batch (expected ~40) |

# Happy Path

1. Decentro processes a batch of ~40 CKYC upload records and fires a web-hook to callback endpoint.
2. System receives web-hook and validates `type == BULK_CKYC_INDIVIDUALS_UPLOAD`.
3. System iterates over all records in `data`.
4. For each record with `status = COMPLETED` and `ckycReferenceId = null`, system resolves `customer_id` from internal DB using `fxcid_key`.
5. System constructs SMS body by substituting `ckyc_id` and `customer_id` into the DLT-registered template customer na.
6. SMS dispatched to customer's registered mobile number (fetched from system using `fxcid_key`).
7. Dispatch result (success / failure) logged to Dynamo DB at per-record level.

---

# Edge Cases

| Case | Handling |
| --- | --- |
| `additionalData` is `null` (record failed validation) | Skip — no comms triggered |
| Customer's phone number not found in system for `fxcid_key` | Log as `FAILED` , skip dispatch |
| SMS API failure for a record | Log as `FAILED` ; rest of batch continues — no retry |
