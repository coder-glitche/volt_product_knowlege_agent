# Custom Ad Hoc Comms — DSP Ops Command Centre

---

# What problem are we solving?

- High-urgency situations (system down, regulatory events, one-off campaigns) need immediate comms capability outside the automated pipeline. All comms today are system-triggered and their is no functionality in the system to generate ad-hoc customer communications
- Ops/compliance/analytics are dependent on the product team for any ad-hoc communications which needs to be sent.
- It is a very time consuming activity for even the product team as of today to send ad-hoc communications as it requires a lot of data crunching, template mapping and script creation
- Some comms depend on non-productised flows (like CIC reporting, approved scripts update), making them impossible to productise

---

# How do we measure success?

- **System Independency** — 100% of batches completed with 0 engineering/product tickets
- **Time to send** (maker → dispatch) — < 10 min
- **No Scripts required** — comms can be triggered directly via the Command Centre(API knowledge independent direct through Command centre of DSP)
- **Unauthorized send rate** — 0% — all batches pass checker before dispatch
- **S3 log coverage** — 100% of sent batches logged with per-record status
- **Batch scale** — Up to 2,00,000 comms with < 2% error
- **Delivery failure rate** — < 2% per batch

# What is the solution?

## Overview

Process for ad-hoc communications - 

1. Operations/Product-ops(comms creator) team create and manage SMS/email templates. Then they share an excel template with all required data fields to be generated by analytics team.
    - Excel generation SOP
        - All comms are driven by a single excel file with a fixed schema.
        - Comms creator prepares and updates the **comms excel** with all **Required fields** as per **Column filling follows SOP** and shares it with Analytics.
        - https://docs.google.com/spreadsheets/d/1Ygp7EKeryMqvjucLGA_L5In14c_raYkTLsDNxUkJJiA/edit?usp=sharing
        - **Column filling follows SOP**
            - Column filling follows SOP:
                - column_names  are lowercase, words separated by underscore.
                - column_names in excel should match the exact variable_names used in sms and email templates.
        - **Required fields**
            
            
            | Channel | Required fields |
            | --- | --- |
            | SMS |   • DLT template ID
              • SMS template body, 
              • variables |
            | Email |   • SendGrid template ID,  
              • variables |
    - Ownership
        1. L1 owner : Operations/Product Ops
        2. L2 owner : Product 
2. Analytics team generates recipient data with relevant details (variables)
    - Data generation SOP
        - Analytics will generate every data column in excel as per the format given by comms creator..
        - Analytics should only provide and identifier_type, Identifier_id(**opportunity ID/ loan account ID)** the **Phone and email address** should be derived by the System(**Tech team**).
3. Operations agent use “Bulk communications” maker in the comms section in the command centre and creates a bulk communication checker task by uploading comms batch file after performing Comms batch file Excel SOP.
- Comms batch file Excel SOP
    - email (Y / N) , sms (Y / N), dlt_template_id, sendgrid_template_id, sms_body_template should be filled by the maker.
    - **The excel should be converted to csv file before uploading to maker nodal.**
    - The CSV must have two mandatory columns:identifier_typeandidentifier_id. All rows in a single CSV must use the sameidentifier_type — eitherFENIX_LOAN_ACCOUNT_IDorFENIX_OPPORTUNITY_ID.
    - PII details should not be filled by the maker. Set them to`$`and the system will fetch them automatically. The PII columns are:
        - phone_number,
        - email,
        - name,
        - pan,
        - date_of_birth,
        - address.
    - For phone_number and email, leaving the cell blank also works — the system treats blank the same as`$`only in case of `phone_number` and `email` .
    - If needed the PII data can be filled by the maker, and that data will be used to trigger the comms.
    - Any additional columns added for template variables (e.g.`amount`,`due_date`) must have a value in every row. A blank or`$`in these columns will mark that row as INVALID.
    - Do not add a`status`column — it is written by the system.
    - The Excel file must be converted to CSV before uploading

# Scope

## In scope

- Once comms creator has the batch file with all recipient details(with Ids, body and variables) of all recipients. He uses the Actions button and selects "Bulk Send Comms" option in Comms section.
- Maker flow
    - recipient batch file upload (xlsx)
    - remarks window
    - Validation flow
        - System auto-validates the uploaded excel before creating a checker task.
        - **Validation checks**
            
            
            | Check | Rule |
            | --- | --- |
            | `validate_column_names` | All column names must be lowercase with words separated by `_`; reject if spaces are found. |
            | `check_template_body_variables` |   • Variable names in `sms_body_template` (e.g. `{{customer_name}}`) must exactly match column names in the sheet else reject
            
              • similar check for email.However `email_body_template`  must be fetched using email template id and send grid API [GET](https://api.sendgrid.com/v3/templates/{sendgrid_template_id})-link.ensure you are providing authorization ****and enable GET access with (bearer token)
            
              • if send grid API didnt send response then its invalid template id so reject |
            | `reject_if_duplicates` | Reject batch if duplicate `opportunity_id` |
            | `check_empty_column_fields rows` | If for the corresponding selected channels if required fields(columns) are missing reject batch and show them as unverified rows in validation window.
            In case of both channels(sms,email) selected if either one has all fields pass. |
            |  |  |
        - Once checks are done Displays count of verified rows and highlights identified errors.
        - Provide option to download the verified file with row level errors of each recipient.
        - Confirm button or cancel button shown for checker task creation.
- Checker flow
    - Under the comms section of command centre.
    - Recipient batch file download
    - Remarks window
    - Accept - Reject buttons
- S3 batch logging: success / failure per record (sync API response)

### **Out of scope**

- Automated retry on record-level failure (logged as failed, no retry)
- Live tracking of click and open events and Database storage only logs on s3

# Happy path

1. OPS/Product OPS creates template → defines fixed columns + variable column names → shares template excel with Analytics
2. Analytics fills in unique_identifier_id (opportunity_id/ loan_account_id) and all variable columns for each recipient row → returns completed excel to OPS(comms creator)
3. Operation agent opens Comms section → clicks Actions → selects **Send Bulk Comm**
4. Maker modal opens → uploads recipient batch file (.xlsx)(following [Comms batch file Excel SOP](Custom%20Comms%20based%20for%20Ad-hoc%20situations%20v2%2032fe8d3af13a80b49c7dc1e2df7abcc2.md)) → file validation runs → reviews verified file  → submits for approval
5. Batch saved as `PENDING_CHECKER_APPROVAL` — no messages sent yet
6. Checker opens approval modal → reviews batch details → downloads recipient file for verification → approves or rejects.
7. On approval, system dispatches comms to all recipients via selected channels.
8. Delivery logs (success / failure ) written to S3 — sync responses, no web hooks.

# Edge cases

| Cases | Handling |
| --- | --- |
| Required field missing on submit | Inline validation, highlight empty fields |
| Checker rejects batch | Status → `REJECTED`, maker notified, no comms sent |
| Record-level send failure | Logged as `FAILED` in S3, rest of batch continues — no retry |
| API error during dispatch at batch level | Batch-level failure logged;  |

    | `timestamp` | Dispatch timestamp |
    | `variables` | Variables used comms |
    | `reciepent address` | phone number, email address |
